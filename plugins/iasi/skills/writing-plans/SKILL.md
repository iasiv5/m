# Writing Plans Skill

这个 skill 用于把已批准的设计或足够清晰的需求，转换成普通 GitHub Copilot 编码 agent 可直接消费的实施计划。

这是一个 planning skill，不是 execution skill。
在这个 skill 中，先写计划，再决定是否开始实现。

## 来源致谢

这个中文化适配版本致敬 obra/superpowers 的 writing-plans skill：
https://github.com/obra/superpowers/tree/main/skills/writing-plans/

当前实现不是逐字搬运，而是面向 GitHub Copilot 的中文化、manual-first、artifact-backed 适配版。

## 何时使用

在以下场景优先使用这个 skill：
- 已经有批准过的设计文档，需要拆成具体实施步骤
- 需求已经足够清晰，但改动不适合直接开写
- 工作会分成多个任务、多个文件或多个验证点
- 需要在编码前把测试、验证命令、任务边界写清楚
- 需要把实现工作交接给后续执行者

## 不要路由到 writing-plans

以下请求默认不要使用这个 skill：
- 还处在需求澄清和方案比较阶段
- 直接实现、改代码、写补丁、跑测试
- 代码审查、交接总结、状态同步
- 简单事实问答或代码解释

如果还没有批准过的设计，优先使用 `brainstorming`。

## 硬约束

- 先写完整计划，再进入实现
- 先规划文件结构和职责，再拆任务
- 每个任务都必须有明确验证；没有验证的任务不算完成
- 禁止占位符：不要写 `TODO`、`TBD`、`later`、`add validation` 这类空话
- 使用精确文件路径；定位优先用“路径 + 符号/章节锚点 + 可选当前行段”，不要把行号当成唯一契约
- checkpoint commit 是建议，不是每个微步骤的硬要求
- 计划写完后先做 inline 自检，再交给用户审阅
- 本 skill 的终态是产出计划并交接执行，不是自己直接开始编码

## PHASE 0: 验证输入

开始前先确认：
- 输入是已批准的设计文档，或至少是足够清晰、能直接拆任务的需求
- 当前工作是否跨多个独立子系统
- 是否已经知道关键路径、受影响文件范围和主要验证方式
- 输入里是否还存在明显的方案未定、关键约束缺失或逻辑矛盾

如果设计仍然覆盖多个独立子系统：
- 先建议拆成多份计划
- 每份计划只覆盖一个可以独立完成和验证的子项目

如果输入仍然存在明显的方案未定、关键约束缺失或逻辑矛盾：
- 明确说明当前输入还不适合直接写计划
- 退回 `brainstorming` 先补设计或澄清边界

## PHASE 1: 读取最小必要上下文

在写计划前，先读取最小必要上下文：
- 设计文档或需求摘要
- 相关文件、测试、命名模式、目录约定
- 已存在的验证命令、测试框架、脚手架模式

目标不是把全仓库看完，而是回答：
- 这次改动应该落到哪些文件
- 哪些命名和边界必须沿用
- 最便宜的验证方式是什么

## PHASE 2: 先规划文件结构

在定义任务前，先列出文件结构和职责：
- 会创建哪些文件
- 会修改哪些文件
- 每个文件承担什么职责
- 哪些文件应该一起变化，哪些边界必须保持稳定

规则：
- 优先跟随现有仓库模式
- 文件按职责拆分，不按想象中的“技术层”硬切
- 如果某个现有文件已经过大，可以在计划里纳入一次与当前需求直接相关的拆分
- 不做和当前目标无关的重构

## PHASE 3: 拆成可执行的小任务

任务要足够小，通常以 2-5 分钟的单步为粒度。

对代码变更，优先使用验证驱动循环：
1. 写失败测试或失败检查
2. 运行并确认当前失败
3. 写最小实现
4. 运行并确认通过
5. 如有必要，建议做一次 checkpoint commit

如果某类工作不适合严格 TDD：
- 仍然要定义“改动前可观察到的失败或缺失”
- 仍然要定义“改动后可观察到的通过条件”

任务必须满足：
- 独立可理解
- 涉及文件明确
- 命令明确
- 预期结果明确

如需固定骨架，先读取 [references/task-skeleton.md](references/task-skeleton.md)。

## PHASE 4: 写实施计划文档

路径选择顺序：
1. 用户显式指定路径
2. 仓库已有计划文档约定
3. 默认写到 `docs/plans/<YYYY-MM-DD>-<feature>-implementation-plan.md`

如需固定骨架，先读取 [references/plan-template.md](references/plan-template.md)。

实施计划至少覆盖：
- 目标
- 架构快照
- 文件结构与职责
- 任务清单
- 每个任务的验证命令与预期结果
- 执行纪律
- 最终验证

## PHASE 5: Inline 自检

写完计划后，读取 [references/plan-self-review.md](references/plan-self-review.md) 并完成单轮 inline 自检。

优先修复会让执行者卡住或把事情做错的问题：
- 设计要求没被覆盖
- 任务仍有占位符
- 后文命名和前文定义不一致
- 命令不存在或验证方式模糊
- 某个任务大到不可执行

不要为了措辞做多轮循环。修完就继续。

## PHASE 6: 用户审阅与执行交接

计划通过自检后，让用户先审阅。

建议使用类似表述：

> 实施计划已写好并保存到 `<path>`。请先确认这份计划；如果没问题，下一步可以按计划由普通 GitHub Copilot 编码流程或人工继续执行。

如果用户要求修改：
- 修改计划
- 重新跑同一轮 inline 自检
- 再次请求审阅

如果用户批准：
- 明确说明这份计划的默认执行方是普通 GitHub Copilot 编码 agent 或人工执行者
- 不要调用不存在的下游 skill
- 不要在本 skill 中直接切入编码

## 执行纪律

把下面的纪律写进计划正文：
- 开始实现前，先批判性复查整份计划；如果发现缺项、矛盾、命名不一致或验证命令无效，先修计划
- 按任务顺序执行，不要无声跳步、合并步或改变任务目标
- 每完成一个任务，都运行该任务定义的验证
- 遇到阻塞、重复失败或计划与仓库现实不符，立即停下来说明，不要猜
- 如果当前就在 `main` 或 `master`，且用户没有明确同意，开始实现前先确认
- 全部任务完成后，运行最终验证并输出修改摘要

## 关键原则

- File structure first：先规划文件与职责，再拆任务
- Bite-sized tasks：任务足够小，便于执行与验证
- No placeholders：不给执行者留下含糊步骤
- Verification required：每个任务都要有验证
- Follow the repo：优先沿用现有仓库结构和命名
- Stop when blocked：遇阻就停，不要猜

## 立即执行

先验证输入是否足够，然后读取最小必要上下文，规划文件结构，拆任务，写计划并完成 inline 自检。
