# Cleanup Skill

这个 skill 用于把已经稳定下来的任务结论，整理成可长期复用的仓库内知识。

它不是中途交接工具，也不是普通总结工具。
它的职责是做终态的知识清理、去重、校正和沉淀。

## 来源致谢

这个适配版本继承了 Khazix `neat-freak` 的核心思想：
- 原始 skill 地址：
  https://github.com/KKKKhazix/khazix-skills/tree/main/neat-freak
- 你是知识编辑，不是记录员
- 长期知识要保持准确、简洁、可复用
- 清理动作要优先考虑删旧、合并和纠偏，而不是一味追加

当前实现不是原版照搬，而是面向 GitHub Copilot 的本地化改造版，重点解决 `handoff` 与长期沉淀之间的职责边界。

## 何时使用

在以下场景使用这个 skill：
- 用户明确要求 `/cleanup`、`cleanup`、`做最终沉淀`、`更新长期知识`、`整理长期记忆`
- 用户明确要求在任务收尾时同步仓库内知识、README、AGENTS 或仓库记忆
- 用户明确要求把已经稳定的任务结论整理成长期知识，而不是只生成下一 chat 的接续上下文

默认按显式长期沉淀意图触发。
仅因为“当前阶段差不多做完了”，不足以自动进入 cleanup。

## 不要路由到 cleanup

以下请求默认不要使用这个 skill：
- 用户要“新开 chat 接着做”“交接到下一轮”“继续当前任务”
- 普通总结、recap、状态同步、进展汇报
- 实现过程中的正常 README、spec、docs、ADR、runbook 更新
- 代码评审、缺陷分析、方案比较、问题排查
- 只是讨论 cleanup skill 本身，而不是要执行一次真实的长期知识沉淀

如果用户只是要把当前工作续接到下一 chat，优先使用 `handoff`。

## 核心边界

- `handoff` 负责会话连续性
- 正式工件更新属于正常开发过程
- `cleanup` 只负责终态长期知识沉淀

同样是改文档：
- 为完成当前任务而改，不算 cleanup
- 为让长期知识系统保持准确、去重、可复用而改，才算 cleanup

## 硬约束

- 优先编辑和清理现有知识，不要把 cleanup 做成流水账追加器
- 先判断任务是否已经稳定，再决定是否执行 cleanup
- 默认只沉淀仓库内长期知识；用户记忆默认不写，除非用户再次明确要求
- 强制 cleanup 也不改变上面的默认边界
- 对仓库事实、正式工件和当前文件状态，优先信任直接读取证据，而不是对话回忆
- 不要把中间态判断、尚未验证的猜测或临时计划写进长期知识
- 不要把普通总结、handoff 内容或一次性叙事直接复制进长期载体

## 三层分工

### 1. 会话连续性层

- 由 `handoff` 承担
- 面向下一 chat
- 可以携带足够继续工作的上下文

### 2. 正式工件层

- 由正常开发流程承担
- 面向当前任务交付
- 包括 spec、plan、README、docs、ADR、runbook、integration guide 等

### 3. 长期知识层

- 由 `cleanup` 承担
- 面向未来会话、未来 agent 和长期维护
- 包括仓库记忆、AGENTS、长期规则、长期知识索引和需长期保留的仓库文档校正

## 触发模型

### 硬触发

仅当用户明确表达长期沉淀意图时，才自动进入 cleanup。

典型表达：
- `/cleanup`
- `cleanup`
- `做最终沉淀`
- `更新长期知识`
- `整理长期记忆`
- `收尾并同步文档和记忆`

### 软提醒

当语义上出现“阶段完成、适合沉淀”的信号时，不自动执行 cleanup，只提示用户现在适合做一次 cleanup。

典型表达：
- `这个阶段做完了`
- `可以交给别人了`
- `新人能直接上手了`
- `准备收尾`

### 反触发

以下语义必须优先路由到 `handoff` 或继续当前执行流程，而不是 cleanup：
- `新开 chat 接着做`
- `帮我交接到下一轮`
- `总结下当前进展让我继续做`
- `继续这个任务`

## 资格门

即使命中了硬触发，也要先判断当前任务是否适合做长期沉淀。

满足以下条件时，才适合执行正常 cleanup：
- 核心目标已经完成，或已停在稳定可交接终点
- 不再依赖下一轮探索来决定主要结论
- 已有正式工件足以承载稳定事实
- 当前做长期沉淀不会把明显的中间态判断写死

如果以上条件不满足，就不要直接进入正常 cleanup。

## 正常 cleanup 流程

### PHASE 0: 验证请求

开始前先确认：
- 用户现在要的是长期知识沉淀，而不是中途续接
- 当前任务已经足够稳定，适合写入长期知识
- 当前知识整理范围主要落在仓库内，而不是用户级全局记忆

如果请求本质上是“继续做”或“交接到下一轮”，改用 `handoff`。

### PHASE 1: 盘点长期知识载体

只读取与当前任务长期沉淀直接相关的最小范围：
- 当前任务涉及的正式工件
- 仓库内 README、AGENTS、docs 或其他长期知识文档
- 仓库记忆（如果当前环境支持）
- 当前任务明确触及的文件和变更事实

如果暂时拿不准 GitHub Copilot 仓库里哪些路径属于长期知识载体，先读取 [references/agent-paths.md](references/agent-paths.md) 再决定扫描面。
不要为了“显得完整”而扫描整个仓库。

### PHASE 2: 判断哪些信息值得长期保留

优先保留这些信息：
- 稳定的边界规则
- 长期会复用的命令、路径、约束和坑点
- 当前仓库中已经形成共识的结构事实
- 未来 agent 或维护者如果不知道就容易做错的内容

优先删除或避免写入这些信息：
- 中间态判断
- 一次性叙事
- 已过期事实
- 能从代码和正式工件直接看出的低价值重复信息

如果不确定“该补到哪里”或“哪些旧信息该删”，先读取 [references/sync-matrix.md](references/sync-matrix.md) 再做取舍。
### PHASE 3: 执行整理

执行顺序优先如下：
1. 删掉过期、重复和互相矛盾的旧知识
2. 合并已有条目，避免并排堆积同义内容
3. 只补真正缺失且已稳定的长期信息

如果这次 cleanup 涉及 GitHub Copilot skill、agent、prompt、instruction、README、metadata 或 CHANGELOG，同步面优先按 [references/sync-matrix.md](references/sync-matrix.md) 检查，不要只改单一入口。

编辑原则：
- 删旧优于追加
- 合并优于堆叠
- 精确优于冗长
- 仓库事实优于会话回忆

### PHASE 4: 输出简洁摘要

完成后只输出简洁同步摘要：
- 改了哪些长期知识载体
- 为什么改
- 哪些东西没有动，为什么没动

不要把 cleanup 输出做成又一份 handoff。

## 强制 cleanup

当用户显式要求 cleanup，但资格门判断任务仍未稳定时：

### PHASE F0: 明确提示风险

必须先明确告诉用户：
- 当前任务仍处于中间态
- 现在直接沉淀可能污染长期知识
- 如果只是想切到下一 chat 继续做，更适合使用 `handoff`

### PHASE F1: 允许用户坚持执行

如果用户在看过风险提示后仍然明确要求继续，可以进入强制 cleanup。

### PHASE F2: 收紧默认写入边界

强制 cleanup 默认只允许更新：
- 仓库内文档
- AGENTS.md
- 仓库记忆
- 其他仓库内长期知识载体

强制 cleanup 默认不写：
- 用户记忆
- 任何跨仓库的长期偏好或全局经验

只有当用户再次明确要求时，才允许把内容写入用户记忆。

## 反膨胀原则

cleanup 的目标不是让知识越来越多，而是让知识越来越干净。

执行 cleanup 时，优先检查：
- 有没有重复条目
- 有没有已经过期但还留着的说明
- 有没有把一次性叙事塞进长期规则文档
- 有没有本该放在正式工件里的内容，被错误堆进长期知识

## 立即执行

先判断这次请求应该走 `handoff`、正常 cleanup 还是强制 cleanup。
如果进入 cleanup，先按需读取 [references/agent-paths.md](references/agent-paths.md) 和 [references/sync-matrix.md](references/sync-matrix.md)，再按最小必要范围读取证据，整理仓库内长期知识，并输出简洁摘要。
